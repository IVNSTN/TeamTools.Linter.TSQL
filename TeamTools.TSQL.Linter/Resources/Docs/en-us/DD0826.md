
# Table has too many columns

|||
|-|-|
| Id | **DD0826**
| Mnemo | TOO_MANY_COLUMNS
| Severity | ℹ Hint
| Category | [DatabaseDesign](./Category_DatabaseDesign.md)
| Source code | [ForeignKeyRecursionRule.cs](../../../Rules/DatabaseDesign/TableHasTooManyColumnsRule.cs)

## Cause

This rule is triggered if a table declaration containing an excessively large number of columns is detected (>80).
<p id="descr">Divide into several tables, set the SPARSE attributes, or define the COLUMNSTORE index.</p>

## Examples

Bad

```sql
CREATE TABLE dbo.foo
(
    id   INT NOT NULL IDENTITY(1, 1) PRIMARY KEY
    , a1 INT NULL
    , b1 INT NULL
    , c1 INT NULL
    , d1 INT NULL
    , e1 INT NULL
    , f1 INT NULL
    ...
    ...
    ...
    , v4 INT NULL
    , w4 INT NULL
    , x4 INT NULL
    , y4 INT NULL
    , z4 INT NULL
    , dt DATE NULL
);
```

Good

```sql
CREATE TABLE dbo.foo
(
    id      INT          NOT NULL IDENTITY(1, 1) PRIMARY KEY
    , title VARCHAR(255) NOT NULL
    , dt    DATE         NULL
);
```

## Tips

💡 If the table has a lot of columns, it is, firstly, similar to an 'god-object', and secondly, it is difficult to maintain. Sometimes wide tables are made specifically, for example, when denormalizing for optimization purposes, but this does not make it more convenient to use such a table. Perhaps the columns lack the SPARSE attribute, or the table itself lacks the COLUMNSTORE index. Or maybe it's just a dump of messy data that should be broken down into several somehow related tables.
