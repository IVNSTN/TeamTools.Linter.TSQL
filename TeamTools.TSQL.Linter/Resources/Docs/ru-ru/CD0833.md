
# Перечисление секций в функции секционирования

|||
|-|-|
| Id | **CD0833**
| Мнемо | PARTITIONING_RANGE_VALUES
| Серьёзность | ⚠ Предупреждение
| Категория | [НепрерывнаяПоставка](./Категория_НепрерывнаяПоставка.md)
| Исходный код | [PartitionFunctionRangeValuesRule.cs](../../../Rules/ContinuousDeployment/PartitionFunctionRangeValuesRule.cs)

## Причина

Правило срабатывает, если в определении функции секционирования перечислены все возможные значения секций.
<p id="descr">Замените перечисление секций на одно значение-заглушку.</p>

## Примеры

Некорректно

```sql
CREATE PARTITION FUNCTION DateRanges (DATETIME2(7))
    AS RANGE RIGHT FOR VALUES
(
    '01/02/2000 00:00:00'
    , '01/03/2000 00:00:00'
    , '01/04/2000 00:00:00'
    , '01/05/2000 00:00:00'
    , '01/06/2000 00:00:00'
    , '01/07/2000 00:00:00'
    , '01/08/2000 00:00:00'
    , '01/09/2000 00:00:00'
    , '01/10/2000 00:00:00'
);
GO
```

Корректно

```sql
CREATE PARTITION FUNCTION DateRanges (DATETIME2(7))
    AS RANGE RIGHT FOR VALUES ('01/02/2000 00:00:00');
GO
```

## Подсказки

💡 Перечисление всех возможных значений в определении функции секционирования не является задачей разработчика. Секции нарезают администраторы и все подробности определения фyнкций секционирования - их обязанность. При применении изменений проекта на сервере обычно игнорируются функции и схемы секционирования. В исходниках чрезмерно длинный список значений не нужен, он ни на что не влияет и только мешает и вводит в заблуждение. Чтобы проект корректно компилировался и раскатывался нужно лишь добавить сам заголовок функции секционирования в проект с одним значением-заглушкой.

💡 Есть схожее правило [HD0916](./HD0916.md). Отличие в том, что это правило реагирует на скрипт определения функции, а HD0916 - на места, где ссылаются на функцию, перечисляя все возможные секции.
